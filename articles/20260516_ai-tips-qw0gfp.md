## はじめに

ChatGPTを使っているけれど、「思った通りの回答が得られない」「毎回質問を考えるのが面倒」と感じていませんか?

本記事では、ChatGPTの回答品質を劇的に向上させる**プロンプトエンジニアリング**の実践的なテクニックを紹介します。エンジニアはもちろん、ビジネスパーソンやクリエイターの方々にも役立つ内容となっています。

**この記事で学べること:**

- よくある失敗例と改善方法

## プロンプトエンジニアリングとは?

プロンプトエンジニアリングとは、AIに対する「指示の出し方」を工夫することで、より精度の高い、意図に沿った回答を引き出す技術です。

従来のプログラミングが「コンピュータへの命令」だとすれば、プロンプトエンジニアリングは「AIとの対話設計」と言えます。

### なぜ重要なのか?

同じ質問でも、聞き方次第で回答の質は大きく変わります。例えば:

**悪い例:**
```
Pythonのコード書いて
```

**良い例:**
```
# 役割
あなたは経験豊富なPythonエンジニアです。

# タスク
CSVファイルを読み込み、特定の列でフィルタリングして
新しいCSVとして出力するPythonスクリプトを作成してください。

# 要件
- pandasライブラリを使用
- エラーハンドリングを含める
- コメントを日本語で記述
- 実行例も含める
```

この差が、AIを「使える」と「使えない」の分かれ道になります。

## プロンプトの黄金パターン

効果的なプロンプトには共通する構造があります。以下の6要素を意識しましょう。

### 1. 役割設定(Role)

AIにどんな専門家として振る舞ってほしいかを明示します。

```
あなたは10年以上の経験を持つフロントエンドエンジニアです。
React、TypeScript、Next.jsに精通しています。
```

**効果:** 回答の視点や専門性が明確になり、より実践的なアドバイスが得られます。

### 2. タスク定義(Task)

具体的に何をしてほしいのかを明確に伝えます。

```
以下のReactコンポーネントをTypeScriptに変換し、
パフォーマンス最適化の提案も含めてください。
```

### 3. 制約条件(Constraints)

守ってほしいルールや制限を指定します。

```
# 制約
- コード量は100行以内
- 外部ライブラリは使わない
- IE11対応は不要
- 説明は簡潔に
```

### 4. 出力形式(Format)

回答の形式を指定することで、後処理がしやすくなります。

```
以下の形式で出力してください:

## 変更内容
- [変更点のリスト]

## コード
```typescript
[コードブロック]
```

## 解説
[簡潔な説明]
```

### 5. 文脈・背景(Context)

タスクの背景情報を提供すると、より適切な回答が得られます。

```
# 背景
現在、Eコマースサイトの商品一覧ページを開発中です。
ユーザー体験向上のため、無限スクロールを実装したいと考えています。
```

### 6. 例示(Examples)

期待する出力の例を示すことで、精度が格段に上がります。

```
# 入力例
"こんにちは、世界!"

# 期待する出力例
{
  "original": "こんにちは、世界!",
  "reversed": "!界世、はちにんこ",
  "length": 9
}
```

## よくある失敗と改善策

実際にプロンプトを使う際の失敗例と対処法を見ていきましょう。

### 失敗例1: 曖昧な指示

**❌ 悪い例:**
```
良いAPIの設計について教えて
```

**問題点:**
- 「良い」の基準が不明
- 対象とする技術スタックが不明
- 回答の深さが定まらない

**✅ 改善例:**
```
# 前提
RESTful APIを設計中です。
Node.js + Expressを使用します。

# 質問
以下の観点でベストプラクティスを教えてください:
1. URLの命名規則
2. エラーレスポンスの形式
3. バージョニング戦略

各項目について、具体例を1つずつ含めてください。
```

### 失敗例2: 一度に詰め込みすぎ

**❌ 悪い例:**
```
認証機能を作りたい。JWT使って、リフレッシュトークンもあって、
ソーシャルログインも対応して、セキュリティも万全で、
しかもパフォーマンスも良くて...
```

**改善策: プロンプトチェーンを使う**

複雑なタスクは分解しましょう。

```
# ステップ1
「JWT認証の基本設計」について相談

→ 回答を得る

# ステップ2
「その設計にリフレッシュトークン機能を追加」

→ 回答を得る

# ステップ3
「セキュリティ面での脆弱性チェック」
```

### 失敗例3: 文脈の欠如

**❌ 悪い例:**
```
このコードのバグを直して

[コードのみ貼り付け]
```

**✅ 改善例:**
```
# 背景
ユーザー登録機能を実装中です。

# 期待動作
メールアドレスの重複チェック後、DBに保存

# 実際の動作
重複していても保存されてしまう

# コード
[コード貼り付け]

# 質問
バグの原因と修正方法を教えてください。
```

## 高度なテクニック: Few-Shot Learning

例示を活用することで、より精度の高い回答を引き出せます。

### Zero-Shot(例示なし)

```
以下のテキストを分類してください:
"新製品の発売が延期されました"
```

### Few-Shot(例示あり)

```
以下の例を参考に、テキストを分類してください。

# 例1
入力: "売上が前年比20%増加しました"
出力: ポジティブ

# 例2
入力: "システム障害が発生しました"
出力: ネガティブ

# 例3
入力: "会議は14時からです"
出力: 中立

# 分類対象
"新製品の発売が延期されました"
```

Few-Shotの方が、意図に沿った分類結果が得られる確率が高まります。

## プロンプトの評価と改善

プロンプトは一度作って終わりではありません。継続的に改善しましょう。

### 評価基準

1. **正確性**: 期待した回答が得られたか
2. **効率性**: 追加の質問なしで完結したか
3. **再現性**: 同じ品質の回答が繰り返し得られるか
4. **汎用性**: 似たタスクに応用できるか

### 改善サイクル

```
1. プロンプト作成
   ↓
2. 実行・結果確認
   ↓
3. 問題点の特定
   ↓
4. プロンプト修正
   ↓
5. バージョン管理
   ↓
(1に戻る)
```

### A/Bテストの活用

同じタスクで複数のプロンプトを試し、最も効果的なものを採用します。

```markdown
# バージョンA(簡潔型)
以下のコードを最適化してください。
[コード]

# バージョンB(詳細型)
以下のコードについて:
1. パフォーマンスボトルネックを特定
2. 最適化案を3つ提示
3. 各案のトレードオフを説明
[コード]
```

両方試して、自分のニーズに合う方を採用します。
