## はじめに - あなたの「わからない」は本当に「わからない」ですか?

エンジニアとして、あるいはビジネスパーソンとして働く中で、「なんかよくわからない」「モヤモヤする」という状態に陥ることはありませんか?

実は、この**「わからない」の正体を明確にできるかどうか**が、問題解決のスピードや学習効率、さらにはチーム内でのコミュニケーション品質を大きく左右します。

本記事では、漠然とした「わからない」を具体的な言葉に変換し、思考の解像度を上げるための実践的な手法を紹介します。これにより、以下のような変化が期待できます:

- 質問の質が上がり、的確な回答を得やすくなる
- 自己学習の方向性が明確になる
- チームメンバーへの説明が論理的になる
- 問題の本質を素早く見抜けるようになる

## 「わからない」の4つのレイヤー

まず、「わからない」には実は複数のレイヤーがあることを理解しましょう。

### レイヤー1: 言葉・用語がわからない

最も表層的な「わからない」です。

**具体例:**
```
❌ 「DDDがわからない」
⭕ 「DDDという単語は聞いたことがあるが、正式名称と基本概念を知らない」
```

このレベルは単純な情報不足なので、検索や書籍で解決可能です。

### レイヤー2: 概念は知っているが使い方がわからない

理論は理解しているが、実践方法が不明な状態です。

**具体例:**
```
❌ 「Reactのカスタムフックがわからない」
⭕ 「カスタムフックの定義方法は理解しているが、
   実際のプロジェクトでどのような粒度で切り出すべきか判断基準がわからない」
```

### レイヤー3: 状況・文脈がわからない

個別要素は理解しているが、全体像や背景が見えない状態です。

**具体例:**
```
❌ 「この設計がわからない」
⭕ 「各クラスの役割は理解できるが、なぜこのような責務分割になったのか、
   ビジネス要件との紐付きがわからない」
```

### レイヤー4: 自分の理解度がわからない(メタ認知の不足)

最も深刻な「わからない」で、何がわかっていて何がわかっていないのか自体が不明な状態です。

```
❌ 「なんとなくモヤモヤする」
⭕ 「実装はできたが、この方法が最適なのか、他の選択肢と比較検討できていない」
```

## ステップ1: 「わからない」を分解する - 5W1Hフレームワーク

漠然とした「わからない」を具体化する最初のステップは、**5W1Hで問いを分解すること**です。

### 実践テンプレート

```markdown
## 現在の状況
- What: 何について理解しようとしているか
- Why: なぜそれを理解する必要があるか
- Where: どの範囲・文脈での理解が必要か
- When: いつまでに理解が必要か
- Who: 誰に対して説明できるレベルを目指すか
- How: どのような方法で理解を深めようとしているか

## わからないポイント
- 具体的にどの部分が不明か
- 前提知識で欠けているものは何か
- 類似概念との違いが不明か
```

### 実例: API設計の場合

**Before:**
「RESTful APIの設計がよくわからない」

**After (5W1H分解):**
```markdown
- What: RESTful APIのリソース設計原則
- Why: 新規プロジェクトでAPI設計を担当するため
- Where: BtoBのSaaS製品における管理画面用API
- When: 2週間後の設計レビューまで
- Who: チームの他のエンジニアに説明できるレベル
- How: 公式ドキュメント、設計書レビュー、実装練習

わからないポイント:
- リソースの粒度をどう決めるか(特に関連エンティティの扱い)
- ネストされたリソースとクエリパラメータの使い分け基準
- バージョニング戦略の選択肢とトレードオフ
```

このレベルまで分解できれば、検索キーワードも明確になり、メンターへの質問も的確になります。

## ステップ2: 既知と未知の境界線を引く

次に重要なのは、**自分が何を知っていて、何を知らないのかを可視化すること**です。

### ナレッジマップの作成

簡単なマインドマップやリストで構いません。以下のような形式で整理します:

```markdown
### 理解できている(✅)
- HTTPメソッド(GET, POST, PUT, DELETE)の基本的な使い分け
- JSONでのデータ構造定義
- ステータスコードの基本(200, 400, 500系)

### 曖昧に理解している(⚠️)
- PUTとPATCHの実務上の使い分け
- 認証・認可の実装パターン
- エラーレスポンスの設計規約

### 全く理解していない(❌)
- HATEOASの実装方法と必要性
- GraphQLとの比較における設計思想の違い
- 大規模サービスにおけるAPI Gatewayの役割
```

この可視化により、**学習の優先順位**が明確になります。

## ステップ3: 「わかった」の基準を定義する

「わからない」を言語化したら、次は**「わかった」状態のゴール設定**が重要です。

### 3段階の理解レベル

**Level 1: 説明できる(Explain)**
- 他人に基本概念を説明できる
- 公式ドキュメントを読んで理解できる

**Level 2: 使える(Apply)**
- 実際のプロジェクトで適用できる
- 基本的なトラブルシューティングができる

**Level 3: 評価できる(Evaluate)**
- 複数の選択肢から最適な方法を選べる
- トレードオフを理解して意思決定できる

### 具体例: Dockerの場合

```markdown
【現在の理解度】Level 1.5
- 既存のDockerfileを読んで概要は理解できる
- docker run, docker buildなどの基本コマンドは使える
- しかし、最適なベースイメージの選択やマルチステージビルドの設計判断はできない

【目標とする理解度】Level 2.5(2週間後)
- プロジェクトに適したDockerfileを一から設計できる
- レイヤーキャッシュを意識した最適化ができる
- docker-composeで開発環境を構築できる
- セキュリティベストプラクティスを適用できる

【学習リソース】
1. 公式ドキュメントのBest Practices
2. 社内プロジェクトのDockerfile読解(3パターン)
3. 個人プロジェクトでの実装練習
```

## ステップ4: 質問をアップグレードする技術

「わからない」を言語化できると、**質問の質が劇的に向上**します。

### 質問の3段階進化

**Level 1: 漠然とした質問**
```
「TypeScriptの型定義がわかりません。教えてください。」
```

**Level 2: 文脈のある質問**
```
「TypeScriptでAPIレスポンスの型定義をしていますが、
オプショナルなプロパティの扱い方がわかりません。
`?`と`| undefined`の違いを教えてください。」
```

**Level 3: 思考プロセスを示した質問**
```
「APIレスポンスの型定義で、以下のように書きました:

interface UserResponse {
  id: number;
  name: string;
  email?: string;  // オプショナル
}

しかし、実際のレスポンスではemailが`null`で返ってくるケースがあり、
型エラーが発生しました。

調査の結果:
- `?`はundefinedのみを許容
- `| null`を追加すればnullも許容される

と理解しましたが、一般的なAPI設計において、
オプショナルフィールドはundefined/null/両方のどれで扱うべきでしょうか?
また、その判断基準があれば教えてください。」
```

### 質問テンプレート

質問する際は以下の構造を意識しましょう:

```markdown
## 背景・目的
[なぜこれを理解する必要があるか]

## 現在の理解
[自分が調べて理解したこと]

## 試したこと
[実際に行った調査・実装]

## 具体的な疑問点
[ピンポイントでわからないこと]

## 期待する回答
[どのレベルの情報が欲しいか]
```

## ステップ5: 継続的な「わからない」の棚卸し

思考の解像度を維持するには、**定期的な振り返り**が不可欠です。

### 週次レビューの実践

毎週末に15分だけ以下を実施します:

```markdown
## 今週の「わからなかった」リスト
1. [技術的な疑問]
2. [設計判断での迷い]
3. [コミュニケーションでの齟齬]

## 解決できたもの
- [内容] → [解決方法] → [学び]

## 持ち越すもの
- [内容] → [次のアクション]

## 新たに発見した「わからない」
- [気づいた知識のギャップ]
```

### デジタルツールの活用

私が実践している方法:

- **Notion/Obsidian**: 「わからないDB」を作成し、タグ付けして管理
- **GitHub Issues**: 個人リポジトリで「学習ログ」として記録
- **日報・週報**: 「今週の学び・疑問」セクションを設ける

```markdown
# わからないDB (Notionテーブル例)

| 疑問内容 | カテゴリ | 優先度 | ステータス | 解決日 |
|---------|---------|--------|-----------|--------|
| gRPCとRESTの使い分け | アーキテクチャ | 高 | 調査中 | - |
| React Queryのキャッシュ戦略 | フロントエンド | 中 | 解決済 | 2024-01-15 |
```

## 実践例: 実際のコードレビューでの応用

最後に、この思考法を実務でどう活用するか、具体例を示します。

### ケース: レビューでのフィードバック

**従来の曖昧なコメント:**
```
「このロジック、ちょっとわかりにくいです」
```

**言語化後の具体的なコメント:**
```
「このメソッドについて確認させてください:

【理解できた点】
- ユーザー情報を取得して加工している
- キャッシュを使って高速化している

【わからない点】
1. なぜfilterとmapを2回に分けているのか?
   → パフォーマンス上の理由があれば教えてください

2. `isActive`の判定ロジックが複雑ですが、
   これはビジネスロジックですか?
   → であれば、ドメイン層に切り出した方が良いかもしれません

3. エラーハンドリングが`console.error`だけですが、
   本番環境での監視はどう考えていますか?

【提案】
メソッドを責務ごとに分割し、それぞれに単体テストを書くのはいかがでしょうか?
```

このレベルの具体性があれば、建設的な議論が可能になります。

## まとめ: 「わからない」は成長のシグナル

本記事で紹介した5つのステップを振り返ります:

1. **「わからない」を分解する** - 5W1Hで具体化
2. **既知と未知の境界を引く** - ナレッジマップで可視化
3. **「わかった」の基準を定義する** - 3段階の理解レベル
4. **質問をアップグレードする** - 思考プロセスを示す
5. **継続的に棚卸しする** - 週次レビューの習慣化
